# /lp-goal — LPを各観点100点まで自走 → 公開まで

**前提**: 壁打ち（`/lp-grilling` または `.claude/skills/lp-grilling`）で `lp-spec.md` が固まっていること。
無ければ実行を止めて壁打ちに戻す。**仕様が無いままビルドしない。**

これは「結果（QA合格＝各観点100点）と検証（サブエージェントの採点）」を固定する closed ループ。
手順を細かく指図せず、**実行 → 検証 → 修正を 100点まで繰り返す**のが肝。

---

## 流れ（drive-to-done）

1. **初回ビルド** — `lp-spec.md` どおりに、画像を `scripts/gen_image.py`（GPT Image 2.0 / `gpt-image-2`）で生成 → `index.html` で機能化。
2. **多観点レビュー（maker ≠ checker・必ずサブエージェント）** — **3 体以上のサブエージェントを並列で召喚**し、各自が担当観点を **0〜100 点で採点 ＋ 具体的な修正指示**を返す。自分（maker）では採点しない。
   - **機能チェッカー** — 全 CTA／モーダル／フォーム／リンク／レスポンシブが実際に動くか（実機・ブラウザで確認）
   - **デザインチェッカー** — トンマナ一貫性・FV の訴求・読みやすさ・余白・画像/図の質・ブランド表現
   - **モバイル&速度チェッカー** — 390/430px で横スクロール無・タップ 44px・固定 CTA・LCP/CLS/INP・崩れ
   - **（4 体目・推奨）コピー&法規制チェッカー** — 訴求の強さ・誇張/景表法/薬機法/ステマ規制・特商法表記
3. **修正** — 各観点の指示を反映する（画像の作り直し＝再生成も含む）。
4. **再採点** — もう一度サブエージェントで採点する。
5. **全観点が 100 点になるまで 2〜4 を繰り返す。** どれかが 100 未満なら、最も低い観点から直す。
6. **公開（必ず質問してから）** — 全観点 100 点になったら、**`AskUserQuestion` で「GitHub Pages と Vercel、どちらで公開しますか？」を質問** → 選ばれた先へデプロイ → 本番 URL が 200 を返すこと・計測タグの発火を確認して報告する。

---

## 参考LPを求めてよい

デザインやトンマナの採点基準が曖昧なときは、**ユーザーに参考 LP の URL を 1〜2 本求めてよい**（`WebFetch` で読み、トンマナの当て先＝デザインチェッカーの基準にする）。壁打ちで参考 URL を得ていればそれを使う。

---

## ルール（厳守）

- **1 発目で LP を作らない。壁打ち未了ならビルド禁止**（`lp-grilling`／`CLAUDE.md` で担保）。
- **採点は必ずサブエージェント（新鮮な目）。** maker が自分で「100点」と言わない。
- **全観点 100 未満で「完成」と言わない。** ゲートが通ってから完了報告。
- **修正はスコープ内（`lp-spec.md`）。** 仕様にない機能・コピーを足さない。コピーの良し悪しは人間が壁打ちで決めたもの。
- ループ中は質問しない（参考 LP と最後の公開先選択だけは聞く）。**取り消せない操作（独自ドメイン・決済・フォーム実連携・外部送信）は止めて確認。**
- **安全上限**: 同じ指摘で 3 周膠着、または 6 周で頭打ちなら、残課題を添えて人間にエスカレーションする（無限ループ防止）。
- **STATE**: `.loop/lp-state.md` に「各観点の点数／残課題／次の一手」を毎周記録（再開用。やり直しではなく再開）。
- **秘密情報をコミットしない**（`OPENAI_API_KEY` は環境変数で渡す）。

---

## 公開（デプロイ）

公開先は **GitHub Pages** か **Vercel**（接続は `reference/connections.md`）。
**最後に必ず `AskUserQuestion` で選ばせる**。選択＝公開の承認とみなすが、独自ドメイン・決済・フォーム実連携は別途人間確認。
